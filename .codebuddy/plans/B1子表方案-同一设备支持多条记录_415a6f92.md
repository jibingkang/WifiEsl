## 产品概述

在现有 WifiEsl 标签管理系统基础上，实现"同一任务内同一设备(MAC)支持多条记录"功能。采用 **B1 子表方案**：保持 `task_devices` 主表不变（保留 UNIQUE(task_id, mac) 约束），新增 `task_device_rows` 子表存储同一设备的多次自定义数据。推送逻辑、display_reply 回执机制基本不变，前端以折叠方式展示多行数据，导入时提示用户选择覆盖或追加。

## 核心功能需求

### 1. 数据库层 - 新增 task_device_rows 子表

- 新建 `task_device_rows` 表，包含 id, task_device_id (FK→task_devices.id), sort_order, custom_data, created_at 等字段
- 提供 CRUD 函数：add/get/update/delete 行数据
- 修改 `get_task_detail` 在返回 devices 时携带 rows 列表

### 2. 后端 API 适配

- 新增子表行数据的 API 端点（增删改查）
- 修改设备自定义数据的写入逻辑：优先写入子表行
- 推送执行时合并主表 custom_data + 子表第一行(sort_order最小)的 custom_data

### 3. 前端表格改造

- DeviceDataTable 的 `:key` 从 `dev.mac` 改为 `dev.id`
- customOverrides 的 key 从 `mac` 改为 `dev.id` 或保持 mac 但支持多行展示
- 设备行支持折叠/展开显示多行子表数据
- 每行子数据可独立编辑、添加、删除

### 4. 导入导出适配

- 导出时：同一 MAC 有多行数据时，每行输出一条记录并附加"行号"列
- 导入时：检测到同 MAC 多行数据，弹出 ElMessageBox 让用户选择"覆盖"（替换所有旧行）或"追加"（保留旧行 + 插入新行）

### 5. display_reply 回执机制（基本不变）

- display_reply 回调仍然按 mac 匹配 task_devices 主表 → 更新 update_status
- 不需要任何改动，这是 B1 方案的最大优势

### 边界情况

- 删除设备时级联删除其所有子表行（ON DELETE CASCADE）
- 同一设备推送只推一次（取 sort_order=0 的子行数据 + 主表默认值）
- 导入时用户未选择覆盖/追加策略时的默认行为

## 技术栈

- **后端**: Python 3.11 + FastAPI + aiosqlite + paho-mqtt
- **前端**: Vue 3 Composition API + TypeScript + Element Plus + Pinia + XLSX (SheetJS)
- **数据库**: SQLite（aiosqlite 异步驱动）
- **实时通信**: MQTT 回执 → DB 更新 → WS 广播 → 前端刷新

## 实现方案

### 策略概述

B1 子表方案的核心思想是：**task_devices 主表保持每任务每设备唯一一条记录（控制状态），新增 task_device_rows 子表存储同一设备的多组自定义数据**。这样：

- 推送逻辑几乎不用改（仍按 mac 遍历推送）
- display_reply 完全不用改（仍按 mac 匹配主表）
- 改动集中在：新建表+CRUD函数、前端折叠组件、导入导出适配

### 关键技术决策

**决策1: 数据存储分工**

```
task_devices 主表:
  - 负责状态管理 (update_status: pending/sent/success/failed)
  - 存储"默认"自定义数据 (custom_data 字段保留作为兜底)
  - display_reply 匹配目标
  
task_device_rows 子表:
  - 存储同一设备的多版本自定义数据
  - sort_order 决定优先级（0 = 最高/默认）
  - 推送时取 sort_order 最小的一行合并到主表 custom_data 上
```

**决策2: 推送时的数据合并逻辑**

```python
# executeTask _push_one 中：
# 1. 取主表 custom_data 作为基底
data = {**default_data}
main_custom = dev.get("custom_data")  # 主表的兜底数据
if main_custom: data.update(main_custom)

# 2. 如果有子表行，取 sort_order=0 的数据覆盖
first_row = await get_first_task_device_row(dev["id"])  # 按 sort_order ASC LIMIT 1
if first_row and first_row.get("custom_data"):
    data.update(first_row["custom_data"])

# 3. 推送合并后的 data
result = await wifi_proxy.apply_template(mac, tid, data, ...)
```

**决策3: 前端 customOverrides 的 key 变更**

- 当前: `customOverrides[mac] = { fieldKey: value }` — 以 MAC 为 key
- B1 后: `customOverrides[devId] = { fieldKey: value }` — 以设备 ID 为 key（因为同 MAC 可能有多条展示行）
- 或者保持 `customOverrides[mac]` 但改为 `customOverrides[mac + '_' + rowIndex]`
- **推荐**: 保持以 mac 为 key 的语义，但在 filteredDeviceTableData 展开阶段将多行拆分为独立行，每行使用复合 key

**决策4: 导入时的覆盖/追加策略**

- 解析完 Excel 后，统计每个 MAC 出现次数
- 对出现多次的 MAC，弹出 ElMessageBox.confirm 让用户选全局策略
- "覆盖": 先 DELETE 该 MAC 的所有子表行，再 INSERT 导入的数据
- "追加": 保留现有子表行，新数据以当前最大 sort_order+N 追加

## 架构设计

### 改造前后数据流对比

```
改造前:
  Excel导入 → importedCustomData[mac] → updateTaskDeviceCustomData(mac, data)
                                                    ↓
                                      task_devices.custom_data (每个MAC只有一条)

改造后:
  Excel导入 → 检测重复MAC → 用户选覆盖/追加
                          ↓ 覆盖              ↓ 追加
            delete rows + insert new    insert new rows (sort_order++)
                          ↓
              task_device_rows (多行) ──→ 推送时取 first row 合并
                                                    ↓
                                      task_devices.update_status (display_reply匹配)
```

```

    │ 1:N

task_device_rows (子表-多行数据) ←── NEW! 存储多版本自定义数据
    │ FK → task_devices.id

```

## 实施细节与注意事项

### 性能考虑

- `get_task_detail` 中 JOIN 查询子表数据（一次查询，不 N+1）
- 推送时 `get_first_task_device_row` 是单条索引查询（需建 idx_rows_tdev_id + idx_rows_sort 索引）
- 导入大量设备时批量插入子表行（用 executemany 而非逐条 insert）

### 兼容性考虑

- 已有任务的 task_devices 记录没有子表行，此时行为等同于改造前（只用主表 custom_data）
- 前端判断 `rows && rows.length > 0` 来决定是否显示折叠按钮
- 导出时兼容无子表行的老数据（不输出"行号"列或该列为空）

### Blast Radius 控制

- display_reply 回调完全不改动（零风险）
- 主表 task_devices 的 UNIQUE 约束不动（零迁移风险）
- 子表是纯增量新增，不影响已有数据
- 前端改动集中在两个 Vue 文件，不影响其他页面

### 错误处理

- 子表操作失败不应影响主表状态流转
- 导入时如果部分行写入失败，给出明确错误提示
- 推送时如果子表数据 JSON 解析失败，降级使用主表 custom_data
